28 // Every base driver MUST have a class interface.
29 //
30 // A class interface is simply a class that defines the API that the driver
31 // must follow. Each method defined in the interface must be declared as a pure
32 // virtual as seen below. Having at least one pure virtual will cause this class
33 // to become an abstract class, meaning it cannot be created on its own.
34 // Abstract classes can only beeinherited by a child class that implements its
35 // pure virtual methods.
36 //
37 // This allows abstract classes to act as a contract to classes that inherit
38 // this interface. Classes that do not implement abstract methods cannot
39 // compile,forcing the developer of that class to implement all of the methods
40 // in the interface.
41 //
42 // This step of creating a driver interface and inheriting it is also extremely
43 // important in allowing the use of polymorphism, which helps in
44 // stubs/mocks/fakes for testing and allows the different higher level device
45 // drivers in the HAL to work across platforms.
46 class ExampleInterface
47 {
48  // Methods, variables, constants and enumerations placed in the interface
49  // should be agnostic to the platform. For example, if you are writing an
50  // interface for a Gpio, you would want a set of methods for setting the
51  // direction (input or output) of the pin or the output voltage level
52  // (high/low). You would expect any MCU to be able to do this. Lets say that
53  // the gpio's of a special chip can toggle at a specific interval based on a
54  // timer. This is something that not many other MCUs would support and thus,
55  // shouldn't be in the interface. Refrain from putting variables of anykind in
56  // the interface
57  public:
58  // Place any MCU agnostic enumeration and constants in the interface. If a
59  // method below needs an enumeration passed to it, that enumeration should
60  // most likely be defined here.
